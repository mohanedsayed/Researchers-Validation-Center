# البحث والقرارات التقنية (Phase 0: Outline & Research)

## القرارات التقنية

- **القرار**: استخدام FastAPI مع SQLAlchemy 2.0 (async).
  **السبب**: يوفر FastAPI أداءً عالياً في معالجة طلبات الويب، ويتكامل بشكل ممتاز مع SQLAlchemy 2.0 عبر تقنية async/await، مما يمنع حظر الخيط الرئيسي (Main Thread) أثناء عمليات قواعد البيانات الثقيلة.
  **البدائل**: تم استبعاد Django نظراً لثقل إطاره مقارنة بالمتطلبات الدقيقة لهذه الواجهة الخلفية، وتم استبعاد Flask لغياب الدعم القوي لعمليات async المدمجة مقارنة بـ FastAPI.

- **القرار**: استخدام Celery و Redis لمهام الخلفية.
  **السبب**: يوفر Celery نظام إدارة مهام متين، مما يجعله مثالياً لتنفيذ مهام التجزئة النصية الثقيلة عبر Ollama دون التأثير على استجابة الـ API، بالإضافة إلى `Celery Beat` الذي سيستخدم لتدمير الجلسات المُنتهية دورياً.
  **البدائل**: تم استبعاد `BackgroundTasks` المدمجة في FastAPI لأنها لا توفر ميزات إعادة المحاولة المتقدمة ولا تتحمل توقف السيرفر.

- **القرار**: الاعتماد على S3 مع تشفير AES-256 محلي قبل الرفع.
  **السبب**: يضمن أن الملفات لا تخزن أبداً كنصوص صريحة على الخوادم حتى وإن تم الوصول إليها بطريقة غير مصرح بها.
  **البدائل**: حفظ الملفات محلياً في مجلدات (مستبعد لتجنب مشاكل قابلية التوسع Horizontal Scaling).

- **القرار**: استخدام Ollama مع نموذج Qwen2.5:7b محلياً.
  **السبب**: لضمان خصوصية البيانات وعدم مشاركة نصوص المستخدمين مع أطراف خارجية (مثل OpenAI)، ونموذج Qwen ممتاز في معالجة اللغة العربية ومهام التجزئة.
